# 数据集设计背景和要求

## 一、为什么需要生成模拟数据集

### 1.1 核心目的

演示 **llm_wiki** 桌面应用的核心价值：

> 让 AI 能理解个人/团队的工作内容——项目背景、决策过程、协作链路、日常沟通——然后像一个了解你团队的同事一样，回答问题、搜索整理、关联线索。

### 1.2 为什么不能直接用真实数据

| 约束 | 说明 |
|------|------|
| 隐私合规 | 真实的工作邮件、会议记录涉及客户信息和内部决策，不能公开演示 |
| 数据量 | 真实数据往往太少（几个人几周聊不出多少条）或太杂乱（噪音多） |
| 可控性 | demo 需要精确知道"正确答案是什么"，以便验证 AI 回答是否正确 |
| 演示效果 | 需要数据恰好覆盖足够丰富的工作场景，能展示多格式检索、多轮对话、关联推理等能力 |

### 1.3 生成模拟数据的意义

通过人工设计一套"真实可信"的工作场景数据，让 demo 观众能够在 5 分钟内：
- 理解产品是做什么的（团队+项目+产品一目了然）
- 感受到 AI 查询的价值（问一个问题，立刻得到有依据的回答）
- 相信这套数据是"真实的"（没有明显的合成感，场景设计符合直觉）

## 二、数据集设计原则

### 2.1 产品选型：简单软件 > 复杂硬件

**原则**：demo 受众用 30 秒理解产品，不需要解释领域背景。

- ❌ 激光雷达、嵌入式摄像头 → 需要解释硬件行业背景
- ✅ 企业内部 AI 知识库助手 → 每个人都是知识工作者，秒懂

**结论**：选 SaaS 软件产品，白领工作场景，普通人都有体感。

### 2.2 团队规模：4-5 人

**原则**：少到能记住每个人的角色，多到能产生足够的协作交互。

- 4 人：角色太少，交互链太短
- 10 人：数据量大到无法在 demo 前准备，也记不住谁是谁
- **5 人**：PM + 产品 + 技术负责人 + 开发 + QA/运营，完整又不冗余

### 2.3 时间跨度：6-7 周

**原则**：足够产生丰富内容，又不至于数据量爆炸。

- 1-2 周：内容太少，展示不开
- 9 个月：数据量太大，生成成本太高
- **6 周**：~200-400 条数据，适合一次 demo 循环

### 2.4 内容格式：多格式混合

**原则**：展示 wiki 的多格式理解能力。

| 格式 | 示例 | 说明 |
|------|------|------|
| 邮件 | 项目启动邮件、客户询价邮件、内部决策邮件 | 异步沟通 |
| IM 聊天 | 技术选型讨论、功能优先级争论 | 即时协作 |
| 会议记录 | Kickoff、M1-M6 里程碑会议 | 正式讨论 |
| 文档 | PRD、技术方案、周报、竞品分析 | 结构化输出 |
| 客户资料 | 需求邮件、竞品对比表、签约合同 | 外部协作 |

### 2.5 双故事线交织

**原则**：单一产品开发线太平，引入销售侧的故事更真实，也更能展示 AI 的关联推理能力。

- **产品开发线**：需求 → 设计 → 开发 → 测试 → 发布
- **客户销售线**：潜在客户 → POC演示 → 谈判 → 签约/拒绝
- **交织点**：客户反馈影响功能优先级、资源争抢影响决策

**交织示例**：
- 青云科技签约后提出飞书集成需求 → Peter 调整功能优先级
- 智远银行试点需求很急 → 团队开会讨论是否接 → 决定等青云结束后再启动

### 2.6 日期锚定机制

**原则**：一套数据，多次 demo，每次都能锚定任意"今天"。

```yaml
anchor_date: "2026-04-28"   # ← 改这一行，就能模拟任意日期

# 所有里程碑存储为 day_offset（相对天数）
Milestone: M6 Launch, day_offset: 0  # 锚定日当天
Milestone: M5 Optimization, day_offset: -14d  # 锚定日 -14 天
```

**好处**：
- 问"今天项目进展如何"→ 答案恰好是 anchor_date 当天的状态
- 问"上周发生了什么"→ wiki 知道哪些内容存在、哪些不存在
- 不会有"数据穿越"问题（不会泄漏未来信息）

## 三、数据内容要求

### 3.1 必须包含的场景

| 场景 | 作用 | 至少包含 |
|------|------|----------|
| 日常异步沟通 | 展示 AI 能理解邮件/文档内容 | 每天 1-2 条邮件或文档 |
| 即时讨论 | 展示 AI 能理解多人讨论的过程 | IM 聊天片段 3-5 条/周 |
| 里程碑会议 | 展示 AI 能关联项目阶段和决策 | M1-M6 各 1 份会议记录 |
| 客户协作 | 展示 AI 能理解外部沟通内容 | 4 个客户各 2-3 封邮件 |
| 内部决策 | 展示 AI 能追踪"为什么做这个决定" | 资源争抢场景 1 个 |
| 周报 | 展示 AI 能整合多人输出 | PM 周报 + 各角色周报 |

### 3.2 避免的内容

- 纯闲聊（与项目无关的内容）
- 敏感话题（政治、宗教等）
- 技术细节过深的硬件文档（不适合 demo 受众）
- 时间穿越的数据（未来日期的内容混入过去时间线）

### 3.3 数据体量估算

```
5 人 × 6 周 × 5 天/周 × ~2 条/天 = ~300 条内容
  ├── 邮件：~40%
  ├── IM 聊天：~30%
  ├── 文档：~15%
  ├── 会议记录：~10%
  └── 其他：~5%
```

## 四、客户故事线设计

### 4.1 4 个客户，覆盖不同阶段

| 客户 | 阶段 | 意愿度 | 资源需求 | 作用 |
|------|------|--------|----------|------|
| 青云科技 | 已签约 | 高 | 低 | 展示签约后服务跟踪 |
| 智远银行 | 试点谈判 | 中 | 高 | 展示资源争抢决策过程 |
| 明日家居 | 评估中 | 中 | 中 | 展示竞品比较场景 |
| 恒通物流 | 意向 | 低 | 低 | 展示跟进无果的情况 |

### 4.2 每个客户的故事线

**青云科技（高意愿，已签约）**
- POC演示 → 签约 → 飞书集成需求
- 代表"成功案例"，但也有新增需求

**智远银行（中意愿，高资源）**
- 提出私有化部署需求 → 内部讨论是否接 → 决定等青云结束后再启动
- 代表"资源有限时的决策权衡"

**明日家居（中意愿，有竞争）**
- 发来竞品对比表 → 销售发送技术方案 → 客户说要等等
- 代表"竞争中如何跟进"

**恒通物流（低意愿，无回复）**
- 行业展会接触 → 发资料后无回复
- 代表"需要持续跟进的线索"

## 五、技术实现要求

### 5.1 目录结构

```
data_generation/
├── 数据集生成模板.txt        # 通用模板，下次生成时填充
├── 数据集生成要求.txt        # 本次项目实例参数
├── 数据集设计背景和要求.md   # 本文档
└── raw/sources/              # 生成的原始数据文件
    ├── email/               # 邮件
    ├── im/                  # IM 聊天记录
    ├── docs/                # 文档（PRD、周报、技术方案）
    ├── meeting/             # 会议记录
    └── customer/            # 客户相关资料
```

### 5.2 文件命名规范

```
格式：[类型]_[日期]_[发送者]_[简要描述]
示例：
  email_2026-03-16_Poky_项目启动通知.txt
  im_2026-03-18_Dave_向量数据库选型讨论.txt
  meeting_2026-03-16_Kickoff会议记录.txt
  doc_2026-03-20_Peter_PRD_v1.0.txt
  customer_2026-04-08_青云科技_POC演示反馈.txt
```

### 5.3 日期计算规则

```
绝对日期 = anchor_date + day_offset

anchor_date: "2026-04-28"   # demo 当天
M6 Launch:   day_offset = 0      → 2026-04-28
M5 Optimize: day_offset = -14    → 2026-04-14
M1 Kickoff:  day_offset = -42    → 2026-03-16
```

## 六、生成流程

1. **确定 anchor_date**（demo 当天）
2. **填充模板**：团队配置 → 产品定义 → 里程碑 → 客户管线 → 内容清单
3. **按时间线生成**：从 M1 启动到 M6 发布，逐周填充
4. **审查数据**：
   - 日期无穿越
   - 人物行为符合角色（PM 管进度，销售跟进客户，技术讨论方案）
   - 决策有前因后果（不要跳跃）
5. **导入 llm_wiki**：放入 raw/sources，触发 ingest
6. **验证查询**：在 anchor_date 当天问几个典型问题，检查回答是否正确

## 七、设计决策历史

| 日期 | 决策 | 原因 |
|------|------|------|
| 最初 | 激光雷达产品 | 最初设想 | 
| → 改为 | 嵌入式 AI 摄像头 | 硬件产品太专业，受众不易理解 |
| → 改为 | 企业内部 AI 知识库助手 | 纯软件，SaaS，白领工作场景人人懂 |
| 最初 | 9 个月时间线 | 最初想模拟完整项目周期 |
| → 改为 | 6 周 | 9 个月数据量太大，demo 场景不需要那么长 |
| 最初 | 10 人团队 | 模拟较完整的公司结构 |
| → 改为 | 5 人 | 10 人数据量太大，记不住角色，demo 不需要那么多人 |
| 新增 | 客户销售线 | 单一产品开发线太单调，加销售线更真实，也能展示 AI 关联推理能力 |
| 新增 | 资源争抢场景 | 展示 wiki 能追踪多方立场、决策过程、前因后果 |

## 八、适用场景

| 场景 | 是否适用 | 原因 |
|------|----------|------|
| 演示 llm_wiki 的检索能力 | ✅ | 多格式内容，多维度查询 |
| 演示 AI 能理解团队工作 | ✅ | 双故事线交织，有决策链路 |
| 演示关联推理能力 | ✅ | 客户反馈 → 优先级调整 → 资源分配 |
| 长期项目跟踪演示 | ✅ | 6 周跨度，anchor_date 可调 |
| 多人协作场景演示 | ✅ | 5 人各司其职，交互丰富 |
| 硬件行业背景演示 | ❌ | 软件产品，不涉及硬件供应链等 |
| 超大数据量演示 | ❌ | 300 条数据，适合中小型演示 |
