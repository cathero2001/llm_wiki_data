# llm_wiki 数据集生成 Prompt

> 本文件是给其他 AI Agent 阅读并执行的数据生成指令。
> 使用前先完整阅读「背景与原则」章节，理解为什么要这样做。
> 然后按「执行步骤」逐步生成数据。

---

## 背景与原则

### 核心目的

生成一套模拟企业工作场景数据，用于演示 **llm_wiki** 桌面应用的核心价值：

> 让 AI 理解个人/团队的工作内容（项目背景、决策过程、协作链路、日常沟通），能够像了解团队的同事一样回答问题、搜索整理、关联线索。

### 为什么生成模拟数据而非真实数据

- **隐私合规**：真实工作邮件、会议记录涉及客户信息和内部决策，不能公开演示
- **数据可控**：demo 需要精确知道"正确答案"，便于验证 AI 回答是否正确
- **演示效果**：数据需要恰好覆盖足够丰富的工作场景，能展示多格式检索、多轮对话、关联推理

### 设计原则（必须遵守）

| 原则 | 要求 | 原因 |
|------|------|------|
| 产品选软件 | SaaS/企业内部工具类产品 | demo 受众 30 秒内理解，不需要领域背景知识 |
| 团队 4-5 人 | PM + 产品 + 技术 + 开发 + QA/运营 | 完整又不冗余，数据量可控 |
| 时间 6-7 周 | anchor_date 往前推 6 周 | ~300 条数据，适合一次 demo，不多不少 |
| 多格式混合 | 邮件 + IM + 文档 + 会议 + 客户资料 | 展示 wiki 多格式理解能力 |
| 双故事线交织 | 产品开发线 × 客户销售线 | 单一开发线太单调，加销售线更真实 |
| 日期锚定 | 所有日期用 day_offset 存储 | 改 anchor_date 一行，整套数据自动重算 |

## 产品定义规则

### 推荐产品类型

✅ **企业内部 AI 知识库助手**（推荐）
- 目标用户：客服、销售、行政、HR 等需要快速查知识的一线员工
- 核心功能：自然语言问答 + 多格式文档检索 + 自动知识同步
- 适用场景：白领工作，人人能懂

✅ **企业协作 SaaS**（如流程审批、文档管理等）

❌ **不推荐**：硬件产品（激光雷达、嵌入式设备等）——需要解释行业背景，demo 受众不易理解

### 产品命名

- 中英双语：中文名 + 英文代号
- 示例：WeBot 企业知识库助手（代号 Iris）
- 代号用于内部文档，代号命名简洁

## 团队配置规则

### 标准 5 人团队

| 角色 | 职责 | 产出类型 |
|------|------|----------|
| 项目经理（PM） | 整体进度、跨部门协调、风险管理、周报汇总 | 启动邮件、周报、会议组织 |
| 产品经理 | 需求分析、产品设计、PRD撰写、竞品分析 | PRD、需求文档、产品设计 |
| 技术负责人 | 技术方案、架构设计、开发分工、代码评审 | 技术方案、会议记录、技术决策 |
| 后端开发 | API 开发、向量数据库、检索引擎 | 开发周报、IM讨论 |
| 前端开发 | 对话界面、文档上传界面 | 开发周报、IM讨论 |
| QA/运营（可选） | 测试用例、发布检查、用户反馈整理 | 测试报告、上线检查清单 |

> 5 人覆盖完整，6 人亦可（加一个专门的测试角色）

## 日期锚定规则

### 核心机制

```yaml
anchor_date: "2026-04-28"   # ← demo 当天，每次生成时指定

project_start: "2026-03-16"  # = anchor_date - 42d（6周前）
```

所有里程碑和事件存储为 `day_offset`（相对于 anchor_date 的天数）：

```
M6 Launch:    day_offset = 0      → 绝对日期 = anchor_date + 0
M5 Optimize:  day_offset = -14    → 绝对日期 = anchor_date - 14d
M1 Kickoff:   day_offset = -42    → 绝对日期 = anchor_date - 42d
```

### 规则

- **数据截止线**：anchor_date 当天的内容存在，4/28 之后的内容不存在
- **禁止时间穿越**：不要在早期数据中泄漏未来信息
- **锚定日重算**：只需改 anchor_date 一行，所有日期自动重算

## 客户管线规则

### 4 个客户，覆盖不同阶段

| 客户 | 阶段 | 意愿度 | 资源需求 | 故事线 |
|------|------|--------|----------|--------|
| 客户A | 已签约 | 高 | 低 | 签约后新增需求，跟进服务 |
| 客户B | 试点谈判 | 中 | 高 | 需求复杂，内部讨论是否接单 |
| 客户C | 评估中 | 中 | 中 | 竞品比较，需要突出差异化 |
| 客户D | 意向 | 低 | 低 | 有兴趣但无回复，需要持续跟进 |

### 客户故事线要求

- 每个客户至少 2-3 封邮件/资料
- 客户反馈需要与产品开发线产生交织（影响功能优先级、资源分配等）
- 至少 1 个"资源争抢"场景：两个客户同时需要资源，团队开会决策

## 内容格式规范

### 6 类内容及比例

| 类型 | 占比 | 示例 | 每条特征 |
|------|------|------|----------|
| 邮件 | ~40% | 启动邮件、周报、客户邮件 | 有 subject、有收件人/发件人、有正文 |
| IM 聊天 | ~30% | 技术讨论、功能争论 | 多轮对话、有表情/语气、碎片化 |
| 文档 | ~15% | PRD、技术方案、周报 | 结构化、有标题层级 |
| 会议记录 | ~10% | Kickoff、评审会 | 有议程、有结论、有行动项 |
| 客户资料 | ~5% | 需求邮件、竞品对比表 | 外部视角、有数据/对比 |

### 文件命名规范

```
格式：[类型]_[日期]_[发送者]_[简要描述].txt

示例：
  email_2026-03-16_Poky_项目启动通知.txt
  im_2026-03-18_Dave_向量数据库选型讨论.txt
  meeting_2026-03-16_Kickoff会议记录.txt
  doc_2026-03-20_Peter_PRD_v1.0.txt
  customer_2026-04-08_青云科技_POC演示反馈.txt
```

## 执行步骤

### 步骤 1：确定 anchor_date

与用户确认 demo 日期，即 anchor_date。

```
anchor_date = "2026-04-28"   # 示例：2026年4月28日 demo
```

### 步骤 2：确定项目参数

填写以下参数：

```yaml
anchor_date: "2026-04-28"
project_start: "2026-03-16"        # = anchor_date - 42d
project_name: WeBot 企业知识库助手   # 产品名称
project_code: Iris                  # 代号
budget: 700万人民币
team_size: 5
duration_weeks: 6
launch_date: "2026-04-28"          # = anchor_date
```

### 步骤 3：生成里程碑

从 project_start 到 launch_date，划分 6 个里程碑：

| 里程碑 | day_offset | 绝对日期 | 目标 |
|--------|------------|---------|------|
| M1 启动 | -42d | 2026-03-16 | 立项、团队组建、需求收集 |
| M2 方案确认 | -28d | 2026-03-31 | 技术方案评审、架构设计冻结 |
| M3 原型完成 | -21d | 2026-04-07 | MVP 开发、首批功能上线内测 |
| M4 测试迭代 | -14d | 2026-04-14 | 内测反馈收集、功能迭代 |
| M5 优化上线 | -7d | 2026-04-21 | 功能和性能优化、准备发布 |
| M6 正式发布 | 0d | 2026-04-28 | WeBot 上线运营、客户支持 |

### 步骤 4：分配客户故事线

基于 4 个客户模板，填充具体公司名称、行业、需求：

- 客户A（已签约）：公司名、行业、签约故事
- 客户B（试点谈判）：公司名、行业、需求点、内部讨论
- 客户C（评估中）：公司名、行业、竞品对比情况
- 客户D（意向）：公司名、行业、跟进无果的情况

### 步骤 5：按时间线生成内容

从 M1 到 M6，逐周生成：

**每周需要包含的内容类型：**

- M1（启动）：启动邮件 + Kickoff 会议记录 + 竞品分析初稿 + IM 技术选型讨论
- M2（方案）：技术方案评审会议 + 技术架构文档 + PRD + IM 选型结论
- M3（MVP）：开发周报 + 客户 POC 演示邀请 + POC 演示记录
- M4（内测）：内测反馈汇总 + 功能优先级讨论 + 签约讨论
- M5（发布准备）：上线检查清单 + 客户需求讨论 + 竞品对比 + 发布前评审
- M6（发布）：上线通知 + 使用反馈 + 试点进展 + 周报

**每周每人至少产出：**
- 邮件 1-2 封
- IM 聊天片段 1-2 条
- 文档（周报/PRD/方案）0-1 份

### 步骤 6：审查数据

生成完成后，检查以下问题：

1. **日期无穿越**：早期数据中没有出现比 project_start 更晚的日期
2. **角色行为一致**：PM 不发技术方案，技术负责人不写 PRD
3. **决策有前因后果**：资源争抢场景有讨论过程，不要直接跳到结论
4. **双线交织**：产品开发线和客户销售线有真实的相互影响

### 步骤 7：导入 llm_wiki

生成完成后，将数据文件放入 llm_wiki 的 `raw/sources/` 目录，触发 ingest。

### 步骤 8：验证查询

在 anchor_date 当天，测试以下典型问题：

- "今天项目进展如何？" → 应回答 M6 发布状态
- "智远银行项目什么时候启动？" → 应回答"等青云结束后"
- "恒通物流有没有回复？" → 应回答"没有回复"
- "向量数据库用的什么？" → 应回答技术选型结论

## 决策历史（参考）

| 阶段 | 决策 | 原因 |
|------|------|------|
| 最初 | 激光雷达产品 | 最初设想 |
| → 改为 | 嵌入式 AI 摄像头 | 硬件产品 demo 受众不易理解 |
| → 改为 | 企业内部 AI 知识库助手 SaaS | 纯软件，白领场景人人懂，数据生成简单 |
| 最初 | 9 个月时间线 | 想模拟完整项目周期 |
| → 改为 | 6 周 | 9 个月数据量太大，demo 不需要那么长 |
| 最初 | 10 人团队 | 想模拟较完整公司结构 |
| → 改为 | 5 人 | 10 人数据量太大，记不住角色 |
| 新增 | 客户销售线 | 单一产品线太单调，加销售线更真实 |
| 新增 | 资源争抢场景 | 展示 wiki 能追踪多方立场、决策过程 |

```
data_generation/
├── prompt.txt              # 本文件，给 Agent 的执行指令
├── 数据集设计背景和要求.md   # 背景参考文档
├── 数据集生成模板.txt        # 通用模板
├── 数据集生成要求.txt        # 本次项目的具体参数

    ├── email/              # 邮件
    ├── im/                 # IM 聊天记录
    ├── docs/               # 文档（PRD、周报、技术方案）
    ├── meeting/            # 会议记录
    └── customer/           # 客户相关资料
```
