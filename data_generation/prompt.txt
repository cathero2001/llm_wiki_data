# llm_wiki 数据集生成 Prompt

> **用途**：给 AI Agent（如 Claude、GPT-4）阅读并执行的数据生成指令。
> **目标**：生成一套模拟企业个人工作数据，用于验证「个人超级记忆系统」在
>         跨文档关联、矛盾检测、Pending 追踪、决策追溯等场景下相对传统 RAG 的优势。
> **可复用**：修改「项目参数」章节中的锚点日期、团队、产品即可生成新数据集。

---

## 一、背景：为什么需要这个数据集

### 1.1 核心概念

「个人超级记忆系统」是 Andrej Karpathy 提出的 **llm_wiki** 概念——一个 AI 驱动的个人知识库，
能自动收录用户日常工作接触的所有信息（邮件、IM、会议记录、文档等），像一个了解你全部
工作上下文的同事一样回答问题、检索关联线索、追踪待办事项。

### 1.2 对比目标

此数据集用于对比两类系统的表现：

| 系统 | 原理 | 预期优势场景 |
|------|------|-------------|
| **llm_wiki**（个人超级记忆） | Agent 主动建索引、跨引用、标矛盾、编译结论页 | 矛盾检测、Pending 追踪、跨文档关联 |
| **传统 RAG** | 向量检索 + LLM 生成 | 简单事实查询、最新信息查询 |

### 1.3 为什么用模拟数据

- **隐私合规**：真实工作数据不能公开
- **答案可控**：必须知道「正确答案」来评估 AI 回答是否准确
- **场景覆盖**：需要精心设计的决策反复、资源冲突等场景来展示系统差异
- **日期可变**：同一套数据需要能锚定到任意 demo 当天

## 二、设计原则（必须遵守）

| 原则 | 要求 | 原因 |
|------|------|------|
| **主角视角** | 所有数据围绕一个主角（产品经理 Peter）组织 | 模拟「个人」的知识库，而非团队维基 |
| **软件产品** | 产品必须是 SaaS/软件，不能是硬件 | 受众 30 秒理解，不需要行业背景知识 |
| **双故事线** | 产品开发线 × 客户销售线，必须有交织点 | 单一产品线太单调，交织能展示关联推理 |
| **决策反复** | 至少 1 个决策经历 ≥2 次变更 | 展示 llm_wiki 的矛盾检测 vs RAG 的困惑 |
| **性能多版本** | 至少 2 个版本的量化指标报告，有达标/未达标变化 | 展示时间线追踪和版本对比 |
| **日期占位符** | 所有日期使用 `{{date_N}}` 占位符（N = 相对 project_start 的天数） | 改 anchor_date 一行，整套数据自动迁移 |
| **文件扁平** | 同一个 phase 的文件全部平铺在一个目录 | 分批导入简单，不需要维护子目录 |

## 四、里程碑规划

以 project_start 为第 0 天，anchor_date = project_start + 42 天，共划分 6 个 milestone：

| Milestone | 天数范围 | 日期范围（示例） | 阶段目标 |
|-----------|---------|-----------------|---------|
| M1 项目启动 | day 0~6 | {{date_0}}~{{date_6}} | Kickoff、需求收集、竞品分析初稿 |
| M2 方案确认 | day 7~13 | {{date_7}}~{{date_13}} | 技术方案评审、架构冻结、PRD v1.0 |
| M3 MVP开发 | day 14~20 | {{date_14}}~{{date_20}} | 核心功能开发、首个客户 POC |
| M4 内测迭代 | day 21~27 | {{date_21}}~{{date_27}} | 内测反馈、签约评审、性能 v1 |
| M5 发布准备 | day 28~34 | {{date_28}}~{{date_34}} | 性能优化 v2、资源争抢决策 |
| M6 正式上线 | day 35~43 | {{date_35}}~{{date_43}} | 上线通知、新客户跟进、性能 v3 |

> 注意：`date_42` = anchor_date 当天，`date_43` = anchor_date 次日（用于上线后的补充邮件）。
> 可增设 `date_49`（项目复盘会议，固定为 5 月 4 日）。

## 五、内容生成规范

### 5.1 文件类型与占比

| 类型 | 文件名前缀 | 约占比例 | 示例 |
|------|----------|---------|------|
| 邮件 | `email_` | ~40% | `email_{{date_9}}_Dave_Pinecone成本超预期.txt` |
| IM 聊天 | `im_` | ~25% | `im_{{date_8}}_两个讨论.txt` |
| 会议记录 | `meeting_` | ~10% | `meeting_{{date_0}}_Kickoff会议记录.txt` |
| 文档 | `doc_` | ~15% | `doc_{{date_9}}_Peter_PRD_v1.0.txt` |
| 客户档案 | `customer_` | ~5% | `customer_{{date_15}}_青云客户档案.txt` |

### 5.2 文件命名规范

```
[类型]_[{{date_N}}]_[发送者/作者]_[简要描述].txt
```

- 文件名中必须用 `{{date_N}}`，不能用实际日期
- 描述用中文，简洁明了
- 总字符数控制在 80 以内

### 5.3 日期占位符规则（最重要）

**所有文件中出现的日期都必须使用 `{{date_N}}` 占位符，无一例外。** 包括：
- 文件名中的日期
- 正文中的日期头（如邮件日期行、会议日期行）
- 正文中引用的其他日期（如"预计4月3日完成"→"预计{{date_18}}完成"）
- 周报中的周期范围（如"4/20—4/26"→"{{date_35}}—{{date_41}}"）
- 性能报告中的测试日期
- 客户档案中关键事件的时间线

**禁止的形式**：
- ❌ `2026-03-25`（ISO 硬编码）
- ❌ `3月25日`（中文硬编码）
- ❌ `3/25`（斜杠硬编码）
- ❌ `2026-04-10（对应 {{date_23}}）`（混合形式）

**允许的形式**：
- ✅ `{{date_9}}`
- ✅ `{{date_11}}~{{date_20}}`（日期范围）

唯一例外：意向度评分如 `6/10`、`7/10`（非日期）。

### 5.4 文件内容格式

每种文件类型的第一行应标识类型：

```
# 邮件格式
发件人：Name <email>
收件人：Name <email>
日期：{{date_N}} HH:MM
主题：xxx
（正文）

# IM 格式
【飞书群聊记录】
群组：xxx
日期：{{date_N}}
参与人：xxx
--- 消息记录 ---
xxx：xxx
（对话）

# 会议记录格式
【会议记录】
会议：xxx
日期：{{date_N}}
时间：H:MM~H:MM
地点：xxx
参与人：xxx
（议程+讨论+结论）

# 文档格式
# 标题
版本/作者/日期：{{date_N}}
（正文）

# 客户档案格式
|【类型】customer记录
|【日期】{{date_N}}
|【公司名称】xxx
|【行业】xxx
（其他字段 + 关键事件时间线）
```

## 六、四条故事线（必须全部包含）

### 6.1 产品开发线

贯穿 M1~M6，覆盖完整的软件开发生命周期：

| 阶段 | 关键事件 | 必要产出 |
|------|---------|---------|
| M1 | Kickoff、方向讨论 | Kickoff会议 + 竞品分析 + PRD v0.5 |
| M2 | 方案冻结、PRD 发布 | 技术评审会 + 架构文档 + PRD v1.0 |
| M3 | MVP 开发 + 首个 POC | 开发周报 + POC准备/演示/反馈 |
| M4 | 内测反馈 + 签约 | 内测问题汇总 + 性能基准 v1（未达标）+ 签约评审会 |
| M5 | 性能优化 + 资源决策 | 性能基准 v2（达标）+ 资源争抢决策会议 |
| M6 | 正式上线 + 复盘 | 上线通知 + 上线庆祝会 + 项目总结 + 性能 v3（超预期）|

### 6.2 决策反复线（⭐ 核心亮点）

**向量数据库选型必须经历三次反复**：
1. **M2**：技术负责人发现方案 A（Pinecone）成本超预期 → 建议换方案 B（FAISS）
2. **M3**：后端测试发现方案 B 私有化部署有隐患 → 建议换方案 C（Milvus）
3. **M4**：客户签约需要私有化 → 正式切换方案 C

每次变更都需要独立的邮件或文档来记录，不要在一份文件里完成——这正是 RAG 困惑而 llm_wiki 清晰的地方。

### 6.3 客户销售线

4 个客户，必须覆盖四种结局 + 与产品线的交织：

| 客户 | 结局 | 与产品线的交织点 |
|------|------|----------------|
| 客户 A | ✅ 签约 | POC 演示→签约→提出新需求（影响 Q2 Roadmap）|
| 客户 B | ⏸️ 延后 | 需求触发资源争抢会议，等产品稳定后再推进 |
| 客户 C | ❌ 放弃 | 竞品对比→输给对手，同时消耗团队精力 |
| 客户 D | 🔄 跟进中 | 新产品上线后新开发的线索，有进展但未定 |

### 6.4 Bug 修复线

至少 6-8 个 Bug 分散在 M2~M6 的周报和邮件中，不同人报告、不同人修复：

| 来源 | 至少包含 |
|------|---------|
| Somi 周报（M2/M3） | 飞书集成丢消息、PDF 解析 Bug、数据库选型反复 |
| Sophie 周报（M3） | 前端 Bug（如移动端布局）、消息丢失修复 |
| Vivi 内测汇总（M4） | 内测阶段发现的问题汇总与修复状态 |
| Dave/技术侧（M2/M4） | 成本问题、技术方案变更 |

## 八、生成后自检清单

生成完毕后，逐项验证：

### 日期一致性
- [ ] `grep -rn '\d{4}-\d{2}-\d{2}' data_template/` 无任何匹配（除 `{{date_N}}` 外）
- [ ] `grep -rn '\d{1,2}月\d{1,2}日' data_template/` 无任何匹配（除 `{{date_N}}` 外）
- [ ] `grep -rn '\d{1,2}/\d{1,2}' data_template/` 只有意向度评分（6/10、7/10 等），无日期

### 逻辑一致性
- [ ] 决策反复链：Pinecone 成本邮件 → FAISS 测试邮件 → Milvus 切换邮件 三个节点全部存在
- [ ] 签约评审会的日期在所有引用处一致（会议文件 + IM 引用 + 上线邮件里程碑）
- [ ] 性能三版本（v1 未达标 / v2 达标 / v3 超预期）全部存在且 P95 值递减
- [ ] 每个客户都有「客户档案」文件，且关键事件时间线与其他文件一致
- [ ] 所有 Day-of-week（如"周一"）与对应 `{{date_N}}` 的星期一致

### 角色一致性
- [ ] 技术决策由 Dave 发出，不由 Peter 或 Sally 发出
- [ ] 销售沟通由 Sally 发出
- [ ] 周报由 Somi（后端）/ Sophie（前端）/ Vivi（运营）分别撰写
- [ ] 会议记录里的参会人与实际角色匹配

### 可 Demo 问法覆盖
- [ ] 「今天项目进展如何？」→ 可回答（有 M6 上线数据）
- [ ] 「向量数据库选型为什么改了这么多次？」→ 可追溯决策链
- [ ] 「我最近有什么待办？」→ 有 pending 文件支撑
- [ ] 「性能达标了吗？」→ 有 v1/v2/v3 对比数据

### RAG 对比测试准备
- [ ] 每个 Phase 都有一个明确的 "RAG 容易出错但 llm_wiki 能答对" 的问题
- [ ] 至少 3 个场景涉及跨文档关联（需要串联 ≥3 个文件才能完整回答）
- [ ] 至少 2 个场景涉及矛盾检测（同一主题在不同文件中表述不一致或逐步演变）

## 九、输出目录结构

生成后的文件应按以下结构组织：

```
data_generation/
├── config.yaml                 # 项目参数配置
├── convert_dates.py            # 日期重算脚本
├── prompt.txt                  # 本文件
├── GUIDE.md                    # 操作指南
├── STORY.md                    # 故事线文档
├── COMPARISON_GUIDE.md         # RAG vs llm_wiki 对比攻略
├── data_template/              # 原始模板（含 {{date_N}} 占位符）
│   ├── phase_m1/               #  6 个 .txt 文件，平铺
│   ├── phase_m2/               #  8 个 .txt 文件，平铺
│   ├── phase_m3/               # 12 个 .txt 文件，平铺
│   ├── phase_m4/               # 22 个 .txt 文件，平铺
│   ├── phase_m5/               # 24 个 .txt 文件，平铺
│   └── phase_m6/               # 22 个 .txt 文件，平铺
└── output/                     # 脚本输出（含实际日期）
    ├── phase_m1/ ~ phase_m6/   # 同结构，日期已替换
```

## 十、与 RAG 对比测试的集成

数据集生成后，配合 `COMPARISON_GUIDE.md` 中定义的 15+ 测试问题，
按 6 个 Round 渐进式测试（Round 1 = M1 only, Round 6 = M1~M6 全量），
评估两类系统在以下维度的差异：

| 任务类型 | llm_wiki 预期 | RAG 预期 |
|---------|-------------|---------|
| 简单事实查询 | ⚠️ 中 | ✅ 强 |
| 跨文档关联 | ✅ 强 | ⚠️ 中 |
| 矛盾检测 | ✅ 强 | ❌ 弱 |
| Pending 追踪 | ✅ 强 | ❌ 弱 |
| 时间线归因 | ✅ 强 | ⚠️ 中 |
| 最新信息查询 | ❌ 中 | ✅ 强 |
