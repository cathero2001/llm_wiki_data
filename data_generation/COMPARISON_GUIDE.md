# RAG vs llm_wiki 对比验证攻略

**项目**：WeBot 企业知识库助手 Demo Dataset
**版本**：v2（新增决策反复链 + 性能基准多版本）
**数据**：6周 / 5人团队 / 92个文件 / 4条主线
**目标**：客观评估两种架构在不同任务类型上的表现差异

---

## 一、对比框架

### 1.1 评测对象

| 系统 | 描述 | 关键特征 |
|------|------|----------|
| **llm_wiki** | Karpathy LLM Wiki 模式，编译型知识库 | Agent 主动建索引/跨引用/标矛盾/编译结论 |
| **RAG（对比组）** | 主流方案：向量检索（ChromaDB + OpenAI embedding）+ LLM 生成 | Query 时从原始文档碎片重新发现 |

### 1.2 数据相同

两套系统使用**完全相同的原始文件**（92个），确保对比的是架构能力差异。

### 1.3 评测模型相同

两套系统使用**相同的 LLM**（如 GPT-4o / MiniMax-M2），排除模型差异干扰。

### 1.4 评测原则

- **不偏袒任何一方**：每个问题都要诚实评估哪个系统更适合
- **区分任务类型**：不同任务类型，适合的方案不同
- **透明评分**：评分标准公开，标注争议点

## 二、任务分类与预期表现

### 2.1 任务分类

根据企业知识库场景的常见查询，将 15 个测试问题分为 5 个类别：

| 类别 | 描述 | 代表问题 |
|------|------|----------|
| **A 跨文档关联** | 需要串联多个来源的信息，形成完整叙事 | "青云签约完整过程" |
| **B 矛盾检测** | 涉及多次讨论、版本演变、决策反复 | "向量数据库选型为什么改了这么多次" |
| **C Pending 状态** | 追踪 open loops、待回复邮件、悬而未决的决策 | "我最近有什么邮件需要处理" |
| **D 时间线与归因** | 按时间顺序还原事件，分析因果关系 | "Bug 修复过程是什么" |
| **E 综合分析** | 需要跨多个维度整合信息，形成结论 | "项目有什么风险" |

### 2.2 各系统预期优势领域

| 任务类型 | llm_wiki 预期 | RAG 预期 | 说明 |
|----------|-------------|----------|------|
| **A 跨文档关联** | ✅ 强 | ⚠️ 中 | llm_wiki 编译型页面有天然优势；RAG 碎片拼接能力有限 |
| **B 矛盾检测** | ✅ 强 | ❌ 弱 | llm_wiki 能标注矛盾和决策演变；RAG 不知道文档在"打架" |
| **C Pending 状态** | ✅ 强 | ⚠️ 中 | llm_wiki 有结构化 pending 状态；RAG 需要 prompt 注入逻辑 |
| **D 时间线与归因** | ✅ 强 | ⚠️ 中 | llm_wiki 编译好时间线；RAG 依赖 chunk 边界和时间排序 |
| **E 综合分析** | ✅ 强 | ⚠️ 中 | llm_wiki 合成页提供结构化结论；RAG 碎片拼接不稳定 |
| **F 简单事实查询** | ⚠️ 中 | ✅ 强 | "PDF 解析 Bug 是谁报告的"——两者差别不大 |
| **G 最新信息查询** | ❌ 中 | ✅ 强 | RAG 直接取最新 chunk；llm_wiki 需要编译过程 |
| **H 大规模知识探索** | ⚠️ 中 | ✅ 强 | RAG 按需检索；llm_wiki 受限于已编译页面 |

## 五、评估指标

每道题从以下 6 个维度打分（1-5分）：

| 指标 | 说明 | 满分 |
|------|------|------|
| **完整性** | 答案是否涵盖所有相关文件的重要信息 | 5 |
| **跨文档关联** | 能否把邮件/IM/会议/文档串联成连贯叙事 | 5 |
| **时间线准确** | 事件顺序是否正确，版本演变是否清晰 | 5 |
| **矛盾感知** | 是否能指出文档间的矛盾或不一致 | 5 |
| **Pending识别** | 是否能正确识别 open loops / pending 状态 | 5 |
| **决策追溯** | 决策背景、选项、结论是否完整 | 5 |

**评分标准**：
- 5分：完全满足，答案精准、全面、有结构
- 4分：基本满足，少量遗漏或不够结构化
- 3分：部分满足，有明显遗漏或错误
- 2分：严重遗漏，答案不完整或不正确
- 1分：无法回答或答非所问

## 七、RAG 对比组搭建步骤

### Step 1：准备文件集

```bash
mkdir -p ~/rag_benchmark/docs
# 全部文件
cp output/phase_m*/sources/* ~/rag_benchmark/docs/
```

### Step 2：向量数据库构建

```python
from langchain_community.vectorstores import Chroma
from langchain_openai import OpenAIEmbeddings
from langchain.document_loaders import DirectoryLoader

loader = DirectoryLoader("~/rag_benchmark/docs/", glob="*.txt")
docs = loader.load()

from langchain.text_splitter import RecursiveCharacterTextSplitter
splitter = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50)
chunks = splitter.split_documents(docs)

embeddings = OpenAIEmbeddings()
db = Chroma.from_documents(chunks, embeddings, persist_directory="~/rag_benchmark/chroma_db")
```

### Step 3：查询脚本（朴素 RAG，不要注入额外逻辑）

```python
def rag_query(question, k=5):
    docs = db.similarity_search(question, k=k)
    context = "\n".join([f"[{d.metadata.get('source', 'unknown')}]\n{d.page_content}" for d in docs])
    prompt = f"""基于以下文档回答问题。如果文档中没有相关信息，说明不知道。
不要胡乱猜测。

文档：
{context}

问题：{question}"""
    response = llm.invoke(prompt)
    return response, docs
```

### Step 2：Agent 处理（核心步骤）
